# DeepSpeed ZeRO 1/2/3 与 Megatron-LM TP/PP：大模型分布式训练的显存账本

我第一次在单张 24GB 的卡上尝试全量微调 Qwen2.5-7B 的时候，天真地算过一笔账：7B 参数，fp16 存储一个参数 2 字节，14GB，24GB 的卡绰绰有余吧？结果 `torch.OOM` 打脸打得很快。后来把账真正算清楚才发现，训练态的显存需求是推理态的**八倍起步**——7B 模型光是"模型状态"就要 112GB，还没算激活值。

这篇文章就是把这本账从头到尾算一遍：显存到底花在哪、DDP 为什么浪费、ZeRO 三个阶段各切掉了什么、Megatron 的 TP/PP 又是在切另一个维度的什么，最后落回我自己在消费级单卡上做 LoRA 微调时这些账怎么算。所有数字都是可以从公式推出来的理论账，不掺没法复现的实测值。

## 一、训练态显存到底花在哪：16Φ 的逐项推导

设模型参数量为 Φ（比如 7B 就是 Φ = 7×10⁹）。主流的混合精度训练（fp16/bf16 计算 + fp32 更新）配 Adam 优化器，每个参数在显存里其实存了**六份东西**：

| 项目 | 精度 | 每参数字节数 |
|---|---|---|
| 参数（前向/反向用） | fp16 | 2 |
| 梯度 | fp16 | 2 |
| 主参数副本（master weights） | fp32 | 4 |
| Adam 一阶动量 m | fp32 | 4 |
| Adam 二阶动量 v | fp32 | 4 |
| **合计** | | **16** |

逐项解释一下为什么一个都省不掉：

- **fp16 参数（2Φ）**：前向和反向传播实际参与矩阵乘的权重。用半精度是为了算得快（Tensor Core）、激活值也小一半。
- **fp16 梯度（2Φ）**：反向传播的产物，精度跟着计算精度走。
- **fp32 主参数（4Φ）**：这是混合精度训练的关键。fp16 的尾数只有 10 位，学习率乘上梯度之后的更新量经常小到在 fp16 里直接下溢归零，参数就"更新不动"了。所以优化器内部必须维护一份 fp32 的主参数，更新在 fp32 上做，再 cast 回 fp16 给下一步前向用。
- **Adam 的 m 和 v（4Φ + 4Φ）**：Adam 要为每个参数维护一阶动量（梯度的指数滑动平均）和二阶动量（梯度平方的滑动平均），都是 fp32。这两项加上主参数，合称**优化器状态（optimizer states），共 12Φ**——占了整本账的四分之三。

所以模型状态总量 = 2Φ + 2Φ + 12Φ = **16Φ 字节**。代入 7B：16 × 7G = **112GB**。这就是 24GB 单卡装不下全量微调的根本原因——差的不是一点半点,是五倍。

账本里还有一个**变量项：激活值（activations）**。它不随参数量走，而是随 `batch_size × seq_len × hidden_size × 层数` 走。Megatron 那篇激活重算的论文给过一个标准 Transformer 层的估算（fp16，不做任何重算）：

```
每层激活 ≈ s·b·h·(34 + 5·a·s/h) 字节
（s=序列长度, b=batch, h=hidden 维度, a=注意力头数）
```

前一项 34sbh 是各处的线性激活，后一项 5·a·s²·b 来自物化的注意力分数矩阵——**和序列长度成平方**，长上下文训练主要就是被它压垮的。好在激活值弹性大：梯度检查点（activation checkpointing）只存每层入口、反向时重算，拿约 1/3 的额外前向时间把这项压到近似 2sbh 级别；FlashAttention 则干脆不物化 s² 的注意力矩阵，把平方项直接抹掉。所以分布式训练的账一般先算死的 16Φ，激活值单独作为可调节项处理。

## 二、DDP 的问题：每张卡一份全量冗余

最朴素的多卡方案是数据并行（DDP）：每张卡放一份**完整**的模型副本，各自吃不同的数据分片，反向之后用 all-reduce 把梯度求平均，再各自做一模一样的参数更新。

DDP 的通信很高效（ring all-reduce 每卡通信量约 2Φ，与卡数无关），但显存上是**纯粹的冗余**：N 张卡，就存了 N 份一模一样的 16Φ。8 张 80GB 的 A100 加起来 640GB 显存，理论上能装 40B 参数的模型状态，但 DDP 模式下每张卡还是只能装 80/16 = 5B。所有卡的优化器状态、梯度、参数都完全相同——这份冗余就是 ZeRO 要下刀的地方。

## 三、ZeRO 三阶段：把冗余一层层切掉

ZeRO（Zero Redundancy Optimizer）的思路很直接：既然 N 张卡存的是同一份东西，那就每张卡只存 1/N，用通信换显存。三个阶段对应三刀，每刀切一类状态。设卡数为 N：

### Stage 1：切优化器状态（12Φ → 12Φ/N)

每卡显存：**2Φ + 2Φ + 12Φ/N**。

每张卡只持有 1/N 的优化器状态，只负责更新自己那 1/N 的参数。反向之后梯度做 reduce-scatter（每卡拿到自己负责那段的完整梯度和），本地更新自己的参数分片，再 all-gather 把更新后的参数广播齐。总通信量和 DDP 的 all-reduce 等价（reduce-scatter + all-gather 本来就是 all-reduce 的两半），**通信没有额外代价**。N 足够大时每卡显存趋近 4Φ，相比 16Φ 省了四分之三——这也印证了前面说的"优化器状态是大头"。

### Stage 2：再切梯度（+2Φ → 2Φ/N）

每卡显存：**2Φ + (2Φ + 12Φ)/N**。

既然每张卡只更新 1/N 的参数,那它也只需要那 1/N 参数的梯度。反向传播过程中，每算完一段梯度就 reduce-scatter 到负责的卡上，本卡随即释放不属于自己的部分。通信量仍然和 DDP 持平。N 很大时每卡趋近 2Φ——只剩 fp16 参数本身。工程上 Stage 2 是最常用的甜点位：显存砍掉近 8 倍，通信不多花一分钱。

### Stage 3：连参数也切（2Φ → 2Φ/N）

每卡显存：**16Φ/N**，理论上模型状态随卡数线性摊薄。

这一刀的代价开始显现。参数被切散后，前向传播算到某一层时，本卡手里没有这层的完整权重，必须先 **all-gather** 把这层参数临时凑齐，用完即弃；反向传播再来一次 all-gather。所以总通信量变成：前向 all-gather（Φ）+ 反向 all-gather（Φ）+ 梯度 reduce-scatter（Φ）= 3Φ，是 DDP 的 **1.5 倍**。而且这些 all-gather 在关键路径上，虽然可以靠预取（prefetch 下一层参数）和计算重叠掩盖一部分，但带宽不够时训练吞吐会明显掉。我的理解是：Stage 3 是"装不下才用"的选项，装得下就停在 Stage 2。

把 7B、8 卡代进去感受一下三个阶段的差距（只算模型状态）：

- DDP：每卡 16 × 7 = **112GB**，A100-80G 都装不下；
- Stage 1：2×7 + 2×7 + 12×7/8 = 14 + 14 + 10.5 = **38.5GB**；
- Stage 2：14 + (2+12)×7/8 = 14 + 12.25 = **26.25GB**；
- Stage 3：112/8 = **14GB**。

也就是说 8 张 40GB 的卡，Stage 1 就够全量微调 7B（还得给激活值留空间，所以实际多半配 Stage 2）；而 DDP 模式下同样 8 张卡一张都跑不起来。切冗余的收益就是这么直接。

### ZeRO-Offload / Infinity：一笔带过

顺着 ZeRO 的逻辑再往下切一层：显存不够，就把状态挪出显存。ZeRO-Offload 把优化器状态和梯度放到 CPU 内存，参数更新在 CPU 上做（Adam 是逐元素操作，CPU 做得动，瓶颈在 PCIe 传输）；ZeRO-Infinity 更进一步，参数本身也可以下放到 CPU 内存甚至 NVMe。代价是每步都要过 PCIe 的几十 GB/s 窄门，速度换容量。后面讲单卡场景时会再回到它。

## 四、Megatron TP：在算子内部切

ZeRO 切的是"状态的存放"，模型的计算图每张卡还是完整走一遍。Megatron-LM 的张量并行（TP）切的是另一个维度：**把单个矩阵乘本身拆到多张卡上算**。

### MLP：先列切、后行切

Transformer 的 MLP 是两个 GEMM：`Y = GeLU(X·A)`，`Z = Y·B`（A 升维、B 降维）。Megatron 的切法很讲究：

- **A 按列切**：A = [A₁, A₂]，每卡算 Yᵢ = GeLU(X·Aᵢ)。关键在于 GeLU 是逐元素非线性，列切之后每卡拿到的 Yᵢ 是最终 Y 的完整的一段列，可以**直接过 GeLU，不需要通信**。如果反过来按行切 A，X·A 的结果是部分和，过 GeLU 之前就必须先 all-reduce（GeLU(a+b) ≠ GeLU(a)+GeLU(b)），通信点多一倍。
- **B 按行切**：B = [B₁; B₂]，每卡算 Zᵢ = Yᵢ·Bᵢ，得到的是 Z 的**部分和**，做一次 all-reduce 相加得到完整 Z。

这样整个 MLP 前向只在末尾有**一次 all-reduce**。

### Attention：按头切

多头注意力天然可切：每个头的 Q/K/V 投影和注意力计算完全独立，把 h 个头分到 N 张卡上（等价于 W_Q/W_K/W_V 按列切），各卡独立算自己那几个头，最后的输出投影 W_O 按行切，同样以一次 all-reduce 收尾。

所以**一层 Transformer 的前向恰好两次 all-reduce**（attention 一次 + MLP 一次），反向再对称来两次，共四次。每次 all-reduce 的量级是 `batch × seq × hidden` 的激活值，而且**没法和计算重叠**——它卡在算子中间，下一步计算就等着这个结果。

这就是 TP 只适合机内 NVLink 的原因：这种高频、同步、在关键路径上的通信，只有 NVLink 的几百 GB/s 带宽 + 微秒级延迟才扛得住。跨机走 IB/以太网（几十 GB/s、延迟高一个量级），每层等四次，吞吐直接崩掉。所以经验法则是 **TP 度 ≤ 单机卡数（typically 8）**。

TP 对显存的收益：参数、梯度、优化器状态、激活值（大部分）都随 TP 度摊薄，且切的是"每一层内部"，不像 PP 有负载均衡问题。有个细节值得一提：LayerNorm、Dropout 这些算子不在 TP 的切分范围内，它们的激活值在每张卡上是完整复制的；Megatron 后来的**序列并行（Sequence Parallelism）**就是把这部分激活沿序列维再切一刀，同时把 all-reduce 拆成 all-gather + reduce-scatter，通信总量不变，把 TP 的激活冗余也抹平了——算是 TP 账本上最后一块补丁。

## 五、PP：按层切，和 bubble 作斗争

流水线并行（PP）按层把模型纵向切成 p 段，每张卡（组）负责几层。通信只发生在相邻 stage 之间传激活值，量小、点对点，**适合跨机**。

问题是流水线的空转（bubble）。最朴素的 GPipe 调度：把一个 batch 切成 m 个 micro-batch 依次灌入，先全部前向、再全部反向。头尾各有一段只有部分 stage 在干活的空窗，bubble 占比：

```
bubble ratio = (p − 1) / (m + p − 1)
```

p = 8、m = 8 时空转 46%，一半算力在发呆。压 bubble 只能加大 m，但 GPipe 调度下**每个 stage 要同时挂着全部 m 个 micro-batch 的激活值**等反向，m 越大激活显存越爆——显存和吞吐直接顶牛。

**1F1B 调度**（PipeDream-Flush，Megatron 默认）解开了这个死结：warmup 之后每个 stage 交替执行"一个前向、一个反向"，某个 micro-batch 的反向尽早开始、激活尽早释放。bubble 占比和 GPipe 相同，但 in-flight 的激活值从 m 份降到**最多 p 份**，与 m 解耦。于是可以放心把 m 开大（经验上 m ≥ 4p，bubble 就能压到 20% 以下），显存不再跟着膨胀。

micro-batch 大小本身也是个权衡：切得太碎，单个 micro-batch 的 GEMM 太小，打不满 GPU；切得太粗，m 上不去，bubble 压不下来。这是个要按模型尺寸和卡数实际扫参的旋钮。

## 六、3D 并行：三把刀怎么组合

千亿级模型的标准打法是三者叠加，分工按通信特性对号入座：

- **TP 放机内**：通信最频繁、最延迟敏感，交给 NVLink。TP 度 = 单机卡数（如 8）。
- **PP 跨机**：只传 stage 边界的激活值，点对点、量小，IB 网络扛得住。
- **DP/ZeRO 放最外层**：把 TP×PP 组成的一个"模型副本"整体复制若干份做数据并行，梯度同步频率低（每步一次），还可以叠 ZeRO-1 把优化器状态在 DP 组内再摊一次。

比如 512 卡训 175B：TP=8（机内）× PP=8（跨 8 台机器）× DP=8（8 个副本）。每个维度都用在它通信模式最舒服的位置上。注意 3D 并行里一般只叠 ZeRO-1 而不是 ZeRO-3——参数已经被 TP/PP 切薄了，再上 Stage 3 的 all-gather 通信不划算。

## 七、落回单卡：我做 7B LoRA 时的账

回到开头那张 24GB 的卡。我最终跑通 Qwen2.5-7B 微调用的是 QLoRA，现在可以用同一本账解释为什么它能行。

**LoRA 改变了账本结构**。冻结主干、只训旁路低秩矩阵之后，设可训练参数量为 φ（r=16 挂 attention 投影层时，φ 大约是 Φ 的 0.5%，约 40M）：

- 主干参数：仍要 2Φ（bf16，前向反向都要用）= 14GB。**这项 LoRA 省不掉**。
- 主干的梯度：不需要——`requires_grad=False`，反向传播只把梯度**流过**它传给旁路，不为它分配 2Φ 的梯度存储。
- 主干的优化器状态：不需要，优化器根本不管它。12Φ = 84GB 直接消失，这才是 LoRA 省显存的大头。
- 旁路的 16φ：40M × 16 ≈ 0.6GB，零头。

所以 LoRA 微调 7B 的模型状态 ≈ 14 + 0.6 ≈ 15GB。24GB 卡上剩 9GB 给激活值和 CUDA 上下文，seq_len 一长（激活值随之线性甚至平方增长）就很紧张——这是我实际卡脖子的地方。

**QLoRA 压的是那 14GB**。既然主干只读不写，就把它量化成 4-bit NF4 存储（0.5Φ ≈ 3.5GB，双重量化再抠一点），计算时逐块反量化回 bf16。模型状态直接掉到 4GB 出头，省下来的空间全部让给激活值和更长的序列。代价是每步反量化的开销，训练比纯 LoRA 慢一点，属于可接受的交换。

实际跑起来还有两个旋钮跟这本账直接相关。一个是梯度检查点：LoRA 场景下激活值反而成了显存大头（模型状态已经被压得很小），开启 checkpointing 后能把 seq_len 或 batch 再往上抬一档，多付的重算时间在单卡上通常划算。另一个是 QLoRA 自带的 paged optimizer：把优化器状态放在可分页的统一内存里，激活值偶发尖峰把显存顶爆时自动换页到 CPU 兜底，避免一个长样本让整个训练崩掉——本质上就是 offload 思想在单卡上的轻量版。

**单卡场景 ZeRO 还有没有意义？** 分两半说。ZeRO 的"Z"——切分（partitioning）——在 N=1 时数学上退化：12Φ/1 还是 12Φ，Stage 1/2/3 的分片本身一张卡无从谈起。但 DeepSpeed 在单卡仍有一个实用价值：**offload**。ZeRO-2 + Offload 能把全量微调的优化器状态（84GB）和梯度挪到 CPU 内存，ZeRO-3 + Infinity 连参数都能下放——理论上 24GB 卡 + 128GB 内存也能全量微调 7B，只是每步都在 PCIe 上搬几十 GB，速度会慢到怀疑人生。所以我的结论是：单卡微调，优先用 LoRA/QLoRA 从账本结构上消掉 12Φ，而不是用 offload 把 12Φ 搬来搬去；只有确实需要全参数更新的场景，offload 才是那个"慢但能跑"的兜底。

## 八、总对比表

| 方案 | 切什么 | 每卡模型状态 | 相对 DDP 通信量 | 通信特征 | 适用位置 |
|---|---|---|---|---|---|
| DDP | 不切（复制） | 16Φ | 1×（2Φ） | 每步梯度 all-reduce，可重叠 | 模型装得下单卡 |
| ZeRO-1 | 优化器状态 | 4Φ + 12Φ/N | 1× | 同 DDP | 几乎白赚，默认可开 |
| ZeRO-2 | +梯度 | 2Φ + 14Φ/N | 1× | 同 DDP | 单纯数据并行的甜点位 |
| ZeRO-3 | +参数 | 16Φ/N | 1.5×（3Φ） | 前反向逐层 all-gather，靠预取掩盖 | 装不下才用 |
| ZeRO-Offload/Infinity | 状态下放 CPU/NVMe | 最低可至 2Φ 以下 | — | PCIe 是瓶颈 | 卡少内存多的兜底 |
| Megatron TP | 算子内部（列切/行切/按头切） | ≈ 16Φ/N | — | 每层前向 2 次 all-reduce，无法重叠 | 机内 NVLink，度 ≤ 8 |
| PP（1F1B） | 按层 | ≈ 16Φ/p | — | 相邻 stage 传激活，量小 | 跨机；m ≥ 4p 压 bubble |
| 3D 并行 | 以上组合 | 三维摊薄 | — | 各维度对号入座 | 百亿参数以上 |

把这本账算熟之后，看任何训练配置都能先在纸上过一遍：16Φ 摆在那里，卡的总显存摆在那里，缺口多大、该在哪个维度切、通信预算够不够，基本没有玄学。OOM 不是运气问题，是账没算平。
